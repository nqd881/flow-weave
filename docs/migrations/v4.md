# Migrate To v4

`flow-weave` v4 reorganizes the public package surface around explicit subpaths.

The runtime model is unchanged, but imports are now stricter and more intentional.

## New Public Entrypoints

- `flow-weave`: main app/runtime entry
- `flow-weave/builder`: advanced fluent-builder APIs
- `flow-weave/decorator`: core flow decorators
- `flow-weave/saga`: saga plugin, saga runtime/types, and saga decorators

## Breaking Changes

### Root No Longer Exports Saga APIs

These move to `flow-weave/saga`:

- `sagaPlugin`
- `SagaDef`
- `SagaExecution`
- `SagaStatus`
- `CompensatorStrategy`
- `StepCompensationAction`
- saga decorators like `Saga`, `CompensateWith`, `CommitPoint`

Before:

```ts
import { FlowWeave, sagaPlugin } from "flow-weave";
```

After:

```ts
import { FlowWeave } from "flow-weave";
import { sagaPlugin } from "flow-weave/saga";
```

### Root No Longer Exports Decorators

Core flow decorators now live in `flow-weave/decorator`.

Before:

```ts
import { Flow, Task } from "flow-weave";
```

After:

```ts
import { Flow, Task } from "flow-weave/decorator";
```

Saga decorators live in `flow-weave/saga`:

```ts
import { Saga, CompensateWith, CommitPoint } from "flow-weave/saga";
```

### Advanced Builder APIs Moved To `flow-weave/builder`

If you import explicit builder types or utilities, use the builder subpath.

Before:

```ts
import { FlowDefBuilder, WeaverBuilder } from "flow-weave";
```

After:

```ts
import { FlowDefBuilder, WeaverBuilder } from "flow-weave/builder";
```

Important:

- normal app code still starts from `FlowWeave` at the root
- `app.weaver()` still gives you the fluent authoring surface
- many applications using only `FlowWeave.create().build().weaver()` will not need builder import changes

### Package Exports Are Now Explicit

`package.json.exports` now defines the public package surface.

Use only the documented entrypoints:

- `flow-weave`
- `flow-weave/builder`
- `flow-weave/decorator`
- `flow-weave/saga`

Do not rely on deep internal paths like:

- `flow-weave/dist/...`
- unpublished internal module paths

## What Did Not Change

- `FlowWeave.create().build()` remains the main app entry
- `app.weaver()` and `app.runtime()` still work the same way
- built flow/runtime behavior is unchanged by the package split
- saga still requires `sagaPlugin`

## Common Migration Examples

### Builder App Code

No change if you only use the app entry:

```ts
import { FlowWeave } from "flow-weave";

const app = FlowWeave.create().build();
const weaver = app.weaver();
```

### Saga Plugin

Before:

```ts
import { FlowWeave, sagaPlugin } from "flow-weave";
```

After:

```ts
import { FlowWeave } from "flow-weave";
import { sagaPlugin } from "flow-weave/saga";
```

### Decorators

Before:

```ts
import { Flow, Task } from "flow-weave";
```

After:

```ts
import { Flow, Task } from "flow-weave/decorator";
```

### Saga Decorators

Before:

```ts
import { Saga, CompensateWith } from "flow-weave";
```

After:

```ts
import { Saga, CompensateWith } from "flow-weave/saga";
```

### Builder Types

Before:

```ts
import { WeaverBuilder } from "flow-weave";
```

After:

```ts
import { WeaverBuilder } from "flow-weave/builder";
```

## Recommended Upgrade Order

1. Update saga imports to `flow-weave/saga`
2. Update decorator imports to `flow-weave/decorator`
3. Update advanced builder imports to `flow-weave/builder`
4. Remove any deep/internal package imports
5. Re-run typecheck and tests
